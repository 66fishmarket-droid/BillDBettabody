SCENARIO: Bill - User Upsert
CANONICAL SOURCE: Make Blueprint JSON (Bill - User Upsert blueprint (1).json)
WEBHOOK LABEL: POST/user/upsert
WEBHOOK URL: https://hook.eu2.make.com/cwxh4f7a7akrfnr9ljilctodqm8355af
PURPOSE: Create or update a client profile row in Google Sheets (Clients sheet).
         If client_id exists -> update row. If not -> add new row.

-----------------------------
SCENARIO FLOW (MODULE SUMMARY)
-----------------------------
Module 1:  Webhook trigger (POST/user/upsert)
Module 10: filterRows - Search Clients sheet column A for client_id (case-insensitive)
Module 8:  SetVariables - Capture existing chronic_contraindications value (chronic_old)
Module 9:  SetVariable2 - Compute merged chronic_contraindications (chronic_new)
Router:
  Branch A (client_found): __ROW_NUMBER__ > 0
    Module 4: updateRow - Update existing Clients row
    Module 6: WebhookRespond - Return action: "updated"
  Branch B (client_not_found): __IMTLENGTH__ < 1
    Module 5: addRow - Insert new Clients row
    Module 7: WebhookRespond - Return action: "created"

-----------------------------
INPUT CONTRACT (JSON BODY)
-----------------------------
Top-level required keys:
- client_id (string) REQUIRED — used to look up existing row; also written to column A
- data (object) REQUIRED — all profile fields nested inside this object

CRITICAL PARTIAL UPDATE RULE:
Bill MUST treat this as a partial update. Only send fields that are being
explicitly set or confirmed by the user. Do NOT send empty strings to overwrite
existing data unless the user has explicitly cleared a field.

EXCEPTION: chronic_contraindications MUST always be sent (see merge logic below).
Send "" if no new chronic condition is being added.

-----------------------------
CONTRAINDICATIONS ARCHITECTURE (CRITICAL)
-----------------------------
injuries (AF) and chronic_contraindications (AE) are READ-ONLY fields on the
Clients sheet. Bill MUST NOT write to them via this webhook.

SOURCE OF TRUTH:
- Active injuries        → Contraindications_Temp sheet (written via Bill - Add Injury)
- Chronic conditions     → Contraindications_Chronic sheet (written via Bill - Add Chronic Condition)

DAILY SYNC:
A scheduled Make scenario reads active records from both contraindication tables,
builds a plain-text summary for each, and writes them back to columns AE and AF.
This keeps the Clients sheet current without Bill making extra API calls.

NEVER send injuries or chronic_contraindications in the upsert payload.
Always send them as omitted or "" — Make will not overwrite them if blank.

-----------------------------
ONBOARDING SEQUENCE (CONTRAINDICATIONS)
-----------------------------
During onboarding Bill handles contraindications in two passes:

PASS 1 — User Upsert:
  Write all profile fields. Always omit or send "" for injuries and
  chronic_contraindications. Never populate these via upsert.

PASS 2 — Contraindication webhooks (run AFTER upsert completes):
  If the user mentions a chronic condition → call Bill - Add Chronic Condition
  If the user mentions an active injury    → call Bill - Add Injury
  Bill may need to make multiple calls if there are multiple conditions.
  Write ALL contraindications before moving to Pass 3.

PASS 3 — Sync webhook (run ONCE after all contraindication writes are complete):
  Call Bill - Sync Contraindications to Client with client_id.
  This immediately updates columns AE and AF on the Clients sheet.
  Bill does NOT need to hold contraindications in working memory after this call —
  the next Load Client Context call will return the updated data.

This same three-pass rule applies mid-coaching, not just onboarding.
Bill MUST always use the dedicated contraindication webhooks — never upsert.
Bill MUST always call the sync webhook once after all contraindication writes.

-----------------------------
PREFERENCES FIELD RULES
-----------------------------
- preferences (column AR) is a STRUCTURED FREE-TEXT field
- It contains named blocks that Bill reads and updates surgically
- The primary block is MOTIVATION_PROFILE (see Bill Instructions Section 2.2)

MOTIVATION_PROFILE block format (stored inside preferences):
  MOTIVATION_PROFILE:
  primary=<feedback_driven|plan_driven|social_identity_driven|outcome_driven>
  secondary=<optional profile or "">
  reinforcers=<semicolon-separated list>
  friction_points=<semicolon-separated list>
  preferred_prompts=<semicolon-separated list>
  exercise_variety_preference=<high|standard|low>
  END_MOTIVATION_PROFILE

CRITICAL SURGICAL UPDATE RULE:
- When updating preferences, Bill MUST only replace the block being changed
- Bill MUST preserve all other content in preferences outside the block markers
- Bill MUST read the current preferences value from client context BEFORE writing
- Bill MUST NOT overwrite the entire preferences field unless performing a full reset
  with explicit user consent

-----------------------------
NOTES FIELD RULES
-----------------------------
- notes (column AS) is a free-text field for coaching-relevant observations
- Appropriate content: session observations, adherence patterns, context flags,
  medication notes where training-relevant (e.g. "On beta blockers — HR response suppressed")
- Medications are NOT a standalone field; include only if training-relevant, in notes

-----------------------------
OUTPUT CONTRACT (JSON RESPONSE)
-----------------------------
Both branches return HTTP 200 with:
- status (string) "ok"
- action (string) "updated" OR "created"
- client_id (string) — echoed from webhook input top-level client_id
- timestamp (string) — ISO datetime

SCENARIO: Bill - Injury to Client Load (Webhook)
CANONICAL SOURCE: Make Blueprint JSON (Bill - Injury to Client Load webhook blueprint.json)
WEBHOOK URL: https://hook.eu2.make.com/8z02z38fdsrmqk06x3v5k56sdu2wroty
WEBHOOK PURPOSE:
On-demand sync of active contraindications from Contraindications_Temp and
Contraindications_Chronic into the Clients sheet for a specific client.
Call this immediately after any Add Injury or Add Chronic Condition webhook
to ensure the Clients sheet reflects current contraindication state without
waiting for the overnight scheduled run.

-----------------------------
WHEN BILL MUST CALL THIS
-----------------------------
Bill MUST call this webhook once after completing ALL contraindication writes
in a session. Specifically:
- After Add Injury (one or more calls)
- After Add Chronic Condition (one or more calls)
- After Update Injury Status (if resolving an injury)

Write ALL contraindications first, THEN call this once at the end.
Do NOT call once per contraindication write — one call at the end is enough.

-----------------------------
SCENARIO FLOW (MODULE SUMMARY)
-----------------------------
Module 16: Webhook trigger — expects client_id
Module 2:  filterRows — Search Clients sheet column A for client_id (limit 1)
Module 6:  filterRows — Search Contraindications_Temp where:
             column A (client_id) = client_id from module 2
             column F (status) = "active" (case-insensitive)
Module 8:  TextAggregator — Aggregate active injuries
             if(__IMTLENGTH__ = 0) → emptystring
             else → date_reported | type | description | expected_duration | notes
             Row separator: ; (semicolon)
Module 10: filterRows — Search Contraindications_Chronic where:
             column B (client_id) = client_id from module 2
             column J (status) = "active"
             FILTER gate: injuries aggregator output must not equal blank template
Module 12: TextAggregator — Aggregate active chronic conditions
             if(__IMTLENGTH__ = 0) → emptystring
             else → condition | severity | affected_system | contraindicated_movements | notes
             Row separator: ; (semicolon)
Module 14: updateRow — Update Clients sheet
             column 30 (AE): chronic_contraindications = {{12.text}}
             column 31 (AF): injuries = {{8.text}}
             row: {{2.__ROW_NUMBER__}}
             FILTER gate: chronic aggregator output must not equal blank template
Module 19: WebhookRespond — Return sync result

-----------------------------
SHEETS USED
-----------------------------
- Clients (read row number, write columns AE + AF)
- Contraindications_Temp (read active injury rows)
- Contraindications_Chronic (read active chronic rows)

-----------------------------
COLUMN MAPPING (CONTRAINDICATIONS_TEMP)
-----------------------------
A: client_id
B: date_reported
C: type
D: description
E: expected_duration
F: status
G: notes (note: column F is status filter, G is notes — aggregator uses col 6 = G)
H: date_resolved

Aggregator output format per injury:
date_reported | type | description | expected_duration | notes

-----------------------------
COLUMN MAPPING (CONTRAINDICATIONS_CHRONIC)
-----------------------------
A: record_id
B: client_id
C: condition
D: severity
E: affected_system
F: contraindicated_movements
G: notes
H: date_added
I: last_reviewed
J: status

Aggregator output format per condition:
condition | severity | affected_system | contraindicated_movements | notes

-----------------------------
EMPTY HANDLING
-----------------------------
Both aggregators use an if(__IMTLENGTH__ = 0) conditional:
- Zero active records → emptystring written to cell (clears it correctly)
- One or more records → pipe-separated summary written to cell
- Additional filter gates on modules 10 and 14 prevent blank template strings
  from being written if the if() conditional doesn't fire correctly

-----------------------------
INPUT CONTRACT (JSON BODY)
-----------------------------
Flat JSON object. One key only:
- client_id (string) REQUIRED

Data structure defined on webhook (Make): client_id (Text, not required)

-----------------------------
OUTPUT CONTRACT (JSON RESPONSE)
-----------------------------
{
  "status": "ok",
  "action": "contraindications_synced",
  "client_id": "<echoed from input>",
  "injuries_updated": "<aggregated injury text or empty string>",
  "chronic_updated": "<aggregated chronic text or empty string>",
  "timestamp": "<ISO datetime>"
}

Bill should read injuries_updated and chronic_updated from the response
and use them as working context for the remainder of the session — no need
to reload full client context just to see the updated contraindications.

SCENARIO: Bill - Injury to Client Load (Scheduled)
CANONICAL SOURCE: Make Scenario (cloned from webhook version, timed trigger)
WEBHOOK PURPOSE:
Overnight scheduled sync of active contraindications from Contraindications_Temp
and Contraindications_Chronic into the Clients sheet for ALL clients.
Runs automatically — Bill does not call this. It is a safety net to ensure
the Clients sheet stays current even if the webhook version was not called.

-----------------------------
SCENARIO FLOW (MODULE SUMMARY)
-----------------------------
No webhook trigger — timed execution only.
Module 1:  filterRows — Search Clients sheet, no filter (returns ALL client rows)
           Acts as iterator — processes one client bundle at a time
Module 2:  filterRows — Search Contraindications_Temp where:
             client_id = current client row client_id
             status = "active" (case-insensitive)
Module 3:  TextAggregator — Aggregate active injuries (same logic as webhook version)
             if(__IMTLENGTH__ = 0) → emptystring
             else → date_reported | type | description | expected_duration | notes
Module 4:  filterRows — Search Contraindications_Chronic where:
             client_id = current client row client_id
             status = "active"
Module 5:  TextAggregator — Aggregate active chronic conditions (same logic as webhook version)
             if(__IMTLENGTH__ = 0) → emptystring
             else → condition | severity | affected_system | contraindicated_movements | notes
Module 6:  updateRow — Update Clients sheet columns AE + AF for current client row

NOTE: Module numbers will differ from the webhook version due to absence of
webhook trigger module. Always verify module references match actual scenario
when making changes.

-----------------------------
BILL BEHAVIOURAL RULES
-----------------------------
- Bill does NOT call this scenario — it is timer-triggered only
- Bill calls the webhook version for immediate same-session updates
- This scenario is a safety net, not the primary sync mechanism
- No webhook URL exists for this scenario — it cannot be called via API

SCENARIO: Bill - UserID Check
CANONICAL SOURCE: Make Blueprint JSON (Bill - UserID Check.blueprint.json)
WEBHOOK PURPOSE:
WEBHOOK URL: https://hook.eu2.make.com/hvsvswhrdfacm7ag4flv1uhpb1nxbigh
Check whether a client_id already exists in the Clients sheet.
Returns exists: true/false. Used as a gate before upsert and
plan generation flows to determine whether to create or update.

-----------------------------
SCENARIO FLOW (MODULE SUMMARY)
-----------------------------
Module 1: Webhook trigger — expects client_id
Module 2: filterRows — Search Clients sheet column A for client_id
          (case-insensitive, limit 1)
Router:
  Branch A (found): __ROW_NUMBER__ > 0
    WebhookRespond — exists: true, client_id from sheet row, row_index
  Branch B (not found): __IMTLENGTH__ < 1
    WebhookRespond — exists: false, client_id echoed from input

-----------------------------
INPUT CONTRACT (JSON BODY)
-----------------------------
Flat JSON object. One key only:
- client_id (string) REQUIRED

No nesting. No other keys referenced.

Fragile bits / must-follow rules:
- filterRows searches Clients column A using case-insensitive text equal
- limit is 1 — only first match returned
- If client_id is missing the filter expression will fail and scenario errors

-----------------------------
OUTPUT CONTRACT (JSON RESPONSE)
-----------------------------
Two possible response shapes:

1) Found (exists: true):
{
  "exists": true,
  "client_id": "<string>",
  "row_index": <number>
}
client_id is returned from the sheet row (not echoed from input).
row_index is the Google Sheets row number from the search module.

2) Not found (exists: false):
{
  "exists": false,
  "client_id": "<string>"
}
client_id is echoed from webhook input.
row_index is NOT included in this branch.

-----------------------------
BILL BEHAVIOURAL RULES
-----------------------------
- Call this before post_user_upsert when it is unclear whether a client
  profile already exists
- If exists: true → proceed with update flow
- If exists: false → proceed with create/onboarding flow
- Never invent or assume a client_id exists without checking first

-----------------------------

  "client_id": "cli_001"
}

PURPOSE:

HOW THE SCORING WORKS (CRITICAL)
-----------------------------
Each exercise is scored across five variables. Each returns 0 or 1.

Scoring variables:

v_match_movement (required_tags.movement_pattern)
  - If movement_pattern NOT sent → auto-scores 1 (pass through)
  - If movement_pattern sent → 1 if exercise matches, 0 if not

v_match_primary_muscles (required_tags.primary_muscles)
  - If primary_muscles NOT sent → scores 0
  - If primary_muscles sent → 1 if exercise matches, 0 if not

v_match_training_focus (required_tags.training_focus)
  - If training_focus NOT sent → scores 0
  - If training_focus sent → 1 if exercise matches, 0 if not

v_match_secondary_muscles (optional_tags.secondary_muscles)
  - If secondary_muscles NOT sent → scores 0
  - If secondary_muscles sent → 1 if exercise matches, 0 if not

v_match_flags (optional_tags.special_flags)
  - If special_flags NOT sent → scores 0
  - If special_flags sent → 1 if exercise matches, 0 if not

v_exclude_special_flags (exclude_tags.special_flags)
  - If special_flags NOT sent → 0 (no exclusion)
  - If special_flags sent and exercise matches → 1 (exercise excluded)

MINIMUM TO GET RESULTS:
If Bill only sends segment_type and nothing else, every exercise scores
1 (movement auto-pass only) — nothing reaches the threshold of 3 and
the response will be empty. Bill MUST send enough tags to generate
meaningful scores. The minimum useful payload includes segment_type
plus at least two of: movement_pattern, primary_muscles, training_focus.

-----------------------------
WEEK DATE CALCULATIONS (FIXED)
-----------------------------
week_start_date formula (module 10):
{{if(7.i - 1 = 0; formatDate(1.plan.week_start; "YYYY-MM-DD"); formatDate(addDays(1.plan.week_start; ((7.i - 1) * 7)); "YYYY-MM-DD"))}}

- Week 1 (i=1): uses plan.week_start directly
- Week N (i=N): plan.week_start + ((N-1) * 7) days
- Example with week_start 2026-02-23:
  Week 1: 2026-02-23
  Week 2: 2026-03-02
  Week 3: 2026-03-09
  Week 4: 2026-03-16

week_end_date: always week_start_date + 6 days (correct).

CRITICAL — plan.week_start MUST ALWAYS BE A MONDAY:
Because Make.com derives every week boundary by adding multiples
of 7 to plan.week_start, a non-Monday start cascades into every
week being non-Monday-aligned. This breaks the Sunday automation
check (which searches Plans_Weeks for start_date = next Monday)
and makes cross-week comparisons unreliable.

RULE: Bill must always send the Monday of the current calendar
week as plan.week_start — never a mid-week date.

HOW TO CALCULATE:
  today_dow = day of week (Monday = 1, Sunday = 7)
  week_start = today - (today_dow - 1) days
  If today IS Monday → week_start = today
  If today is Wednesday (dow=3) → week_start = today - 2 days
  If today is Sunday (dow=7) → week_start = today - 6 days

STARTING MID-WEEK IS FINE — use Monday as week_start anyway:
A client who starts training on Wednesday Feb 26 gets week_start
= Feb 24 (Monday). Bill simply does not create sessions for Feb
24–25 (days already past). Sessions begin from Feb 26 onward
within Week 1. All subsequent week boundaries are Monday-aligned.
This is the "start today, but the week grid is always Monday"
convention.

Module 2:  SetVariables — normalise and trim all input fields; apply defaults
Module 3:  SetVariables — compute dedupe_key, context_append string, email subject
Module 4:  filterRows — search Issue_Log for open matching row using dedupe key
           Matches on: issue_type (col D) + entity_type (col E) +
                       client_id (col B) + entity_id (col F)
           All comparisons case-insensitive. Limit 1.
Router:
  UPDATE branch (match found — __ROW_NUMBER__ > 0):
    Module 6:  updateRow — update existing Issue_Log row
    Module 8:  ActionSendEmail — notify on update
    Module 10: WebhookRespond — return action: "updated"
  CREATE branch (no match — __IMTLENGTH__ < 1):
    Module 7:  addRow — insert new Issue_Log row
    Module 9:  ActionSendEmail — notify on create
    Module 11: WebhookRespond — return action: "created"

-----------------------------
INPUT CONTRACT (JSON BODY)
-----------------------------
Flat JSON object. No nesting.

Required:
- issue_type (string) REQUIRED — category of issue e.g. "data_error", "coaching_gap",
                                  "missing_field", "safety_flag", "session_anomaly"
- entity_type (string) REQUIRED — what the issue relates to e.g. "session", "step",
                                   "client", "block", "week", "exercise"
- description (string) REQUIRED — plain English description of the issue

Optional (send as "" if not applicable):
- client_id (string) — defaults to "no_client" if missing or blank
- source (string) — defaults to "bill" if missing or blank; other values: "system", "user"
- entity_id (string) — the ID of the specific entity e.g. session_id, step_id, block_id
- context (string) — additional context; appended with timestamp and source into
                     the context column. Format stored: "[YYYY-MM-DD HH:mm:ss] source: context"
- suggested_action (string) — recommended next step for whoever reviews the log
- priority (string) — e.g. "low", "medium", "high", "critical"
- status (string) — defaults to "open" if missing or blank; use "resolved" to close

-----------------------------
DEDUPLICATION LOGIC
-----------------------------
Make builds a dedupe key from:
  lower(client_id + "|" + issue_type + "|" + entity_type + "|" + entity_id)

If a row exists in Issue_Log matching all four fields AND the row is found
by filterRows (no status filter — matches open OR closed rows), it is updated.
If no match, a new row is created.

IMPORTANT: Deduplication matches regardless of current status. A "resolved"
issue with matching fields will be updated not recreated. If you want a
genuinely new issue for the same entity, change entity_id or issue_type.

-----------------------------
UPDATE BEHAVIOUR (when existing row found)
-----------------------------
- description: preserved from original row (not overwritten)
- context: existing context + "\n" + new context_append (appended)
- suggested_action: preserved from original if new value is blank
- status: overwritten with incoming value if non-blank; otherwise preserved
- priority: overwritten with incoming value if non-blank; otherwise preserved
- timestamp, client_id, source, issue_type, entity_type, entity_id: always overwritten

-----------------------------
COLUMN MAPPING (Issue_Log)
-----------------------------
Col A (0):  timestamp
Col B (1):  client_id
Col C (2):  source
Col D (3):  issue_type
Col E (4):  entity_type
Col F (5):  entity_id
Col G (6):  description
Col H (7):  context
Col I (8):  suggested_action
Col J (9):  status
Col K (10): priority

-----------------------------
OUTPUT CONTRACT (JSON RESPONSE)
-----------------------------
Create response:
{
  "status": "ok",
  "action": "created",
  "row_index": "<new row number>",
  "timestamp": "<YYYY-MM-DD HH:mm:ss>"
}

Update response:
{
  "status": "ok",
  "action": "updated",
  "row_index": "<existing row number>",
  "timestamp": "<YYYY-MM-DD HH:mm:ss>"
}

-----------------------------
BILL BEHAVIOURAL RULES
-----------------------------
- Use this for genuine data or coaching issues that need human review —
  not as a general logging mechanism for routine operations
- Always include entity_id when the issue relates to a specific record —
  this is critical for deduplication to work correctly
- context should explain what Bill observed or what triggered the issue,
  not repeat the description
- suggested_action should be actionable for a human reviewer
- For safety flags (e.g. client reports chest pain, dizziness): set
  priority "critical" and source "bill"
- Do not use this to log successful operations — only genuine issues
