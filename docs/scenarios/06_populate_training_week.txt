SCENARIO: Bill - Populate Training Week
CANONICAL SOURCE: Make Blueprint JSON (Bill - Populate Training Week blueprint (2).json)
WEBHOOK LABEL: Bill - Training Week Actions
WEBHOOK URL: https://hook.eu2.make.com/2vs9htbixx68m2hdbxinro9tdp55arao
PURPOSE:
Write session details and steps to Google Sheets for a full training week.
For each session: updates the existing Plans_Sessions row with session metadata
and session_summary. For each step in each session: inserts a new row into
Plans_Steps with full exercise prescription and pattern fields.
Called once per week after the Full Training Block Generator has created the
skeleton session rows.

-----------------------------
SCENARIO FLOW (MODULE SUMMARY)
-----------------------------
Module 1:  Webhook trigger — receives full week payload
Module 6:  BasicFeeder — Session Iterator (loops over sessions array)
Module 10: filterRows — Find Session Row in Plans_Sessions by session_id
Module 11: updateRow — Update Plans_Sessions row with session metadata
Module 7:  BasicFeeder — Step Iterator (loops over steps array per session)
Module 8:  addRow — Write step to Plans_Steps
Module 12: BasicAggregator — Aggregate step count across all sessions
Module 13: WebhookRespond — Return success response

-----------------------------
INPUT CONTRACT (JSON BODY)
-----------------------------
Top-level required keys:
- client_id (string) REQUIRED
- context (object) REQUIRED
- sessions (array) REQUIRED

context object — required keys:
- plan_id (string) REQUIRED
- block_id (string) REQUIRED
- week_id (string) REQUIRED
- week_number (number) REQUIRED

IMPORTANT: week_id and block_id are generated by the Full Training Block
Generator scenario upstream. Bill MUST NOT invent these — they must come
from the active context loaded at session start.

-----------------------------
SESSIONS ARRAY
-----------------------------
Each item in sessions is a session object.

REQUIRED per session:
- session_id (string) REQUIRED
  CRITICAL: used to find the existing Plans_Sessions row. If no matching
  row is found, the updateRow will fail silently for that session.
- session_summary (string) UI-CRITICAL
  Displayed on the PWA home screen as the client's preview of the upcoming
  session. Must be 2-4 sentences in Bill's voice: plain-spoken, honest about
  effort, warm but direct. Never generic. Never empty.
  If missing or empty: Make will write a blank cell and flag it in the
  response. Bill must treat a missing session_summary as a coaching error
  and regenerate before the client sees their plan.
- steps (array) REQUIRED
  Must be an array. Can be empty but then nothing is written to Plans_Steps.

OPTIONAL per session (missing fields write blank cells — do not omit if known):
- week (number)
- day (number)
- phase (string)
- location (string)
- focus (string)
- exercises (string) — comma-separated exercise name summary
- macros (string)
- supplements (string)
- notes (string)
- session_date (string) — format YYYY-MM-DD
- session_day_of_week (string)
- session_global_number (number)
- intended_intensity_rpe (string or number)
- intended_hr_zone (string) — MUST be a string e.g. "2", never an integer
- estimated_duration_minutes (number)
- linked_sport (string)

TYPE RULES — common mistakes that fail validation:
- intended_hr_zone: always send as a quoted string e.g. "2", never bare integer 2
- duration_value: always send as a number e.g. 600, never a quoted string "600"
- sets, reps, rest_seconds, step_order: always numbers, never strings

FIELDS MAKE WRITES AUTOMATICALLY — DO NOT SEND:
- session_status — hardcoded "Populated" by Make
- week_id, block_id, client_id — Make reads these from the matched
  Plans_Sessions row (module 10), not from input

-----------------------------
STEPS ARRAY
-----------------------------
Each item in steps is a step object.

REQUIRED per step:
- step_order (number) REQUIRED
- segment_type (string) REQUIRED — ONLY valid values: warmup / main / cooldown
- exercise_name (string) REQUIRED — must be canonical name from Exercise Library

CORE FIELDS (send all that apply — missing fields write blank cells):
- step_type (string) — e.g. strength, cardio, mobility, skill
- duration_type (string) — time / distance / reps
- duration_value (number) — seconds or metres depending on duration_type
- target_type (string) — e.g. hr_zone, rpe, pace
- target_value (string)
- sets (number)
- reps (number)
- load_kg (number)
- rest_seconds (number)
- notes_coach (string) — Bill's instructional voice. Cues, intent, safety.
                         Draw from Exercise Library. Depth matched to
                         client detail_level_preference.
- notes_athlete (string) — NEVER written by Bill. Leave empty always.

FIELDS MAKE WRITES AUTOMATICALLY — DO NOT SEND IN STEPS:
- step_id — generated by Make as st_<session_id>_<timestamp>_<step_order>
- session_id — pulled from matched Plans_Sessions row (module 10, col 6)
- week_id — pulled from matched Plans_Sessions row (module 10, col 4)
- block_id — pulled from matched Plans_Sessions row (module 10, col 1)
- client_id — pulled from matched Plans_Sessions row (module 10, col 2)
- status — hardcoded "planned" by Make
- metric_key — auto-populated by Google Sheets formula from exercise_name
- metric_context_key — see metric_context_key rules below

-----------------------------
PATTERN FIELD RULES BY EXERCISE TYPE
-----------------------------
If a pattern_type is assigned, ALL required fields for that pattern MUST
be populated. If required fields cannot be meaningfully populated, do NOT
assign a pattern_type — use a flat prescription instead.

STRENGTH (pattern_type: straight_sets / progressive_load / wave_load / etc.)
MANDATORY: Every strength step (step_type: strength) MUST have a pattern_type.
Use straight_sets as the minimum. Use progressive_load for compound movements
where load increases across sets. Never leave pattern_type blank on a strength step.

Required fields (ALL must be populated when step_type is strength):
- pattern_type (string) — REQUIRED, no exceptions for strength steps
- sets (number)
- reps OR reps_pattern (reps_pattern overrides reps when sets differ)
- load_start_kg (number) — starting load; use load_kg value if flat
- load_increment_kg (number) — explicitly 0 if flat load across sets
- load_peak_kg (number) — highest load reached; same as load_start_kg if flat
- rest_seconds (number)
- tempo_pattern (string) — 4-digit format e.g. "3010"
  Default: "3010" for beginners, "2020" for general population
  Omit ONLY if medically contraindicated or explicitly not appropriate
- rpe_pattern (string) — comma-separated RPE per set e.g. "7,7,8"
Optional:
- tempo_per_set_pattern (string) — if tempo varies per set
- pattern_notes (string) — progression instructions, autoregulation rules

CARDIO — STEADY STATE (pattern_type: steady_state)
Required:
- pattern_type: "steady_state"
- duration_type (string) — "time" or "distance"
- duration_value (number) — seconds or metres
- intensity_start (string) — zone, RPE cue, or pace
- intensity_end (string) — same as intensity_start if not progressive;
  include if effort changes across the duration

CARDIO — INTERVALS (pattern_type: intervals)
Required:
- pattern_type: "intervals"
- interval_count (number) — total number of work intervals
- interval_work_sec (number) — duration of each work interval in seconds
- interval_rest_sec (number) — duration of each rest interval in seconds
- intensity_start (string) — effort target for work intervals
- intensity_end (string) — effort target for final intervals if progressive;
  same as intensity_start if flat effort

CALISTHENICS / BODYWEIGHT (pattern_type: straight_sets / amrap)
Required:
- pattern_type (string)
- sets (number)
- reps OR reps_pattern (string)
- rest_seconds (number)
- rpe_pattern (string) — bodyweight work still needs RPE targets
- tempo_pattern (string) — especially important for bodyweight to
  control time under tension
Optional:
- load_start_kg (number) — include if weighted (e.g. weighted pull-up)
- load_increment_kg / load_peak_kg — if progressive

MOBILITY / WARMUP STEPS
Pattern fields generally not required. Use:
- duration_type + duration_value for timed holds
- sets + reps for movement-based drills
- notes_coach for execution cues — this is the primary field for mobility
No pattern_type needed unless structurally repeating intervals.

-----------------------------
METRIC_CONTEXT_KEY RULES
-----------------------------
metric_context_key is auto-populated by Google Sheets formula for most
exercises. Bill only needs to send it in one specific case:

SEND metric_context_key when:
- The step is a specific distance or time trial where performance will
  be compared against a personal best
- Examples:
  - Running 5K time trial → metric_context_key: "5k"
  - Rowing Erg 2000m for time → metric_context_key: "2000m"
  - Freestyle Swim 100m → metric_context_key: "100m"

DO NOT send metric_context_key when:
- Step is general cardio training (duration-based, steady state)
- Step is a strength exercise (metric_key handles this via sheet formula)
- Example: "Run 25 minutes easy" → no metric_context_key

BILL BEHAVIOURAL RULES
-----------------------------
- NEVER call populate_training_week in the same turn as generate_training_plan.
  After generate_training_plan and load_client_context succeed, you MUST stop
  tool use, return a text response to the client summarising the proposed week
  (session count, dates, focus for each session), and end your turn. Wait for
  the client to explicitly say yes before calling populate_training_week in a
  subsequent message. This is a hard rule — do not complete the full chain in
  one turn regardless of what the client originally asked for.
  Example closing line: "Here's what I've got for Week 1 — [summary]. Want me
  to go ahead and lock this in?"
- Always verify session_ids exist in context before calling — these must
  come from the active week loaded via Load Client Context, never invented
- Never invent plan_id, block_id, week_id — always from loaded context
- session_summary is UI-CRITICAL — treat a missing one as a blocker,
  not an optional field
- notes_athlete MUST always be sent as empty string — never write to it
- Do not send step_id, session_id, week_id, block_id, client_id inside
  step objects — Make handles all of these from the session row lookup
- If pattern_type is assigned, all required pattern fields for that type
  MUST be populated — do not assign pattern_type and leave required fields
  blank
- metric_context_key should only be sent for specific distance/time trial
  steps — not for general training steps
